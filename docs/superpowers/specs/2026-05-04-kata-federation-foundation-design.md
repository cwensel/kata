# Federation Foundation — Instance Identity, Event UIDs, and Origin Provenance

> **Status:** design / spec. Companion to `docs/superpowers/specs/2026-04-29-kata-design.md` (master design), `docs/superpowers/specs/2026-05-03-kata-issue-identity-design.md` (v1→v2 issue identity, this spec's predecessor), and `docs/superpowers/specs/2026-05-03-kata-jsonl-roundtrip-design.md` (the cutover machinery). This spec is the **last identity-layer change before sync work begins** — it locks the durable schema fields that a future sync protocol will depend on, without designing the protocol itself. Schema version bumps from `2` to `3`.

## 1. Locked decisions

These five decisions are settled here and are not re-litigated by the implementation plan.

1. **`meta.instance_uid` — one ULID per kata installation, set at first init or v2→v3 cutover, never changes.**
   - Stored as a single row in the existing `meta` table (`INSERT INTO meta(key, value) VALUES ('instance_uid', '01J…')`).
   - Generated by `uid.New()` at first daemon init OR by the v2→v3 cutover for an existing v2 DB. Once written, never overwritten — there's no API or CLI that mutates this value. A "clone this DB to a new instance" use case is handled at import time (§5.2), not by mutating an existing DB.
   - Used as: (a) the default `origin_instance_uid` for events and purge_log rows written by this daemon; (b) the deterministic tiebreaker in any future sync conflict resolution; (c) loop-detection identity ("is this event one I authored?").
   - Not a row-level identifier and not joinable. It identifies the **kata installation**, not any individual record.

2. **`events.uid` — ULID NOT NULL UNIQUE, generated at insert. Stable identity, NOT a global ordering cursor.**
   - Same pattern as `issues.uid` and `projects.uid` from v2: `uid.New()` at insert time, `uid.FromStableSeed(seed, created_at)` at cutover backfill.
   - This is the stable cross-instance **identity** of an event — the value the sync protocol uses to ack ("I have applied event UID X"), to dedupe on receive, and to reference in tombstones or back-references. It is **not** a global ordering cursor: ULID lexicographic order reflects per-clock time, and once events from multiple instances are merged, lexicographic UID order does not reflect causal or arrival order. The ordered feed remains `events.id` (per-instance AUTOINCREMENT, monotonic on each daemon) plus the synthetic purge cursor; a future sync protocol will use the integer log on the receiving end with UIDs as anchors that the receiver maps to its own ordered position.
   - The integer `events.id` stays as a local ordering anchor and join key — same justification as `issues.id` keeping its integer FK alongside `issues.uid` (issue-identity design §1.2): hot read paths (event polling, SSE replay) want integer ordering, not 26-char string ordering.

3. **`events.origin_instance_uid` — TEXT NOT NULL, defaults to local `meta.instance_uid` at insert.**
   - Set by the daemon's insert path on every event write — reading the local `meta.instance_uid` once at startup and stamping every event with that value. This is the same shape as `events.project_identity` (a denormalized field set on insert).
   - **Deliberately NOT nullable.** A NULL would mean "this event predates federation"; we avoid that branch in every future sync consumer by backfilling the v2→v3 cutover with the cutover-generated `meta.instance_uid`. Every event has an origin from day one of v3.
   - Constraint: `CHECK (length(origin_instance_uid) = 26)`.
   - Not foreign-keyed to anything (no `instances` table — instance identity is a property carried in records, not a referential entity).

4. **`purge_log.uid` and `purge_log.origin_instance_uid` — same shape as events, mirrored.**
   - `purge_log.uid` is `TEXT NOT NULL UNIQUE` generated at purge time via `uid.New()`. Identifies the tombstone itself.
   - `purge_log.origin_instance_uid` is `TEXT NOT NULL` set from local `meta.instance_uid` at purge time. CHECK constraint mirrors events.
   - These two fields make `purge_log` rows the authoritative tombstone feed for federation: a peer ingesting purge_log rows knows (a) which tombstone is which (uid), (b) where it came from (origin_instance_uid), and (c) what to delete locally (the existing `issue_uid` / `project_uid` from v2). No separate tombstones table is needed.

5. **Authority model: hub-authoritative.**
   - This spec doesn't implement sync — but the schema choices presume hub-authoritative resolution because anything more complex (logical clocks, vector clocks, CRDTs) would need additional schema (per-row clocks, replica state) that we are deliberately not adding.
   - Sync protocol design (a separate future spec) will use: hub holds the canonical state; spokes push local writes to hub then pull updates from hub; conflicts at the hub resolve via `updated_at` last-writer-wins with `origin_instance_uid` as deterministic tiebreaker.
   - Per-row UIDs on `comments` and `issue_labels` are **deliberately not added** here. Comments and labels are owned by their issue and addressed through event records (`issue.commented`, `issue.labeled`) which carry stable event UIDs. If the sync protocol design later proves this awkward, those columns can be added via a future cutover.

The rest of this document expands what these decisions imply for the schema, the JSONL contract, the cutover, the wire envelopes, and the test plan.

## 2. Goal

Lock the durable identity columns that any reasonable sync protocol will need, in one v2→v3 cutover, before more users adopt kata. The shape is forward-locked but the protocol itself is not — a future spec will design `kata sync`, the merge-import path, conflict resolution, and the HTTP endpoints, consuming the columns this spec adds.

The downstream goal is that when sync work begins, the schema is already federation-ready: every event and every tombstone carries an authoritative origin and a stable cross-instance identifier, with no further migration required.

## 3. Scope

**In scope**
- Schema additions (edits to `0001_init.sql`):
  - `meta.instance_uid` (singleton row, set at init or cutover)
  - `events.uid` (NOT NULL UNIQUE, ULID)
  - `events.origin_instance_uid` (NOT NULL, ULID)
  - `purge_log.uid` (NOT NULL UNIQUE, ULID)
  - `purge_log.origin_instance_uid` (NOT NULL, ULID)
- Schema-version bump from `2` to `3` (via `currentSchemaVersion` constant).
- Daemon insert paths for events and purges populate the new columns automatically.
- v2→v3 cutover fill rules in the JSONL importer.
- JSONL wire-shape extensions: a new `meta` record kind value (`instance_uid`) and the new fields in `event` / `purge_log` envelopes.
- An optional `--new-instance` mode for `kata import` that **keeps the target identity written by `db.Open`** rather than overwriting it with the source's `meta.instance_uid` — for the "clone this DB to a new machine" use case (§5.2).
- API exposure: `GET /api/v1/instance` returns the local `meta.instance_uid` (so a future spoke client can discover its peer's identity).
- Wire-shape updates: `EventEnvelope`, `PurgeLogEntry` JSON shapes gain `uid` and `origin_instance_uid` fields. Existing fields stay.

**Out of scope (deferred)**
- The sync protocol itself: `kata sync` CLI, push/pull HTTP endpoints, merge-import / UID-keyed upsert apply path, peer state tracking (last-pushed/last-pulled cursors). All deferred to a future spec.
- Per-row UIDs on `comments`, `issue_labels`, or any other table not listed above (per §1.5).
- Logical clocks, vector clocks, or any non-wall-clock ordering. The eventual sync protocol uses `updated_at` + `origin_instance_uid` tiebreaker.
- Cross-instance authentication. Covered by the existing shared-server-mode spec (Phase 2 readiness pass).
- Per-instance tracking on `projects` or `issues`. The sync protocol may need to know which instance authored which issue for conflict resolution; if so, that's an additive cutover later. We don't speculate here.

## 4. Architecture summary

The change is small enough that "architecture" overstates it: five additive schema fields, two of which require a one-time fill rule at cutover, and a handful of insert-path updates in the daemon to stamp every new event and every purge with the local `meta.instance_uid`.

The daemon reads `meta.instance_uid` once at startup into a Go-level `daemon.LocalInstance` value and uses it as the default `origin_instance_uid` for every subsequent insert. There is no per-request lookup.

`uid.New()` is used to generate new event/purge UIDs at insert time (random monotonic — same pattern as v2's issue/project UIDs). `uid.FromStableSeed(seed, t)` is used at cutover backfill to give existing v2 events and purge_log rows deterministic UIDs derived from their `created_at` and a row-stable seed (event id + project, purge id + project).

Soft-delete and purge semantics are unchanged from master spec §3.5. The new `purge_log` fields are populated at purge time alongside the existing fields; the v2→v3 cutover backfills them for existing rows.

## 5. Schema declaration

`internal/db/migrations/0001_init.sql` is the single canonical schema (per JSONL spec §6.0). The federation-foundation changes are **edits to that file**, not a new migration. The `currentSchemaVersion` constant in `internal/db` bumps from `2` to `3`; the JSONL roundtrip's first-boot cutover (per JSONL spec §6.3) is what rolls existing v2 databases forward.

### 5.1 Edits

```sql
-- meta: no schema change. The instance_uid value is seeded by db.Open after
-- init (see §5.2 below). 0001_init.sql does not seed it.

-- events: add uid (NOT NULL UNIQUE) and origin_instance_uid (NOT NULL)
CREATE TABLE events (
  id                    INTEGER PRIMARY KEY AUTOINCREMENT,
  uid                   TEXT NOT NULL UNIQUE,           -- NEW (v3)
  origin_instance_uid   TEXT NOT NULL,                  -- NEW (v3)
  project_id            INTEGER NOT NULL REFERENCES projects(id),
  project_identity      TEXT NOT NULL,
  issue_id              INTEGER REFERENCES issues(id),
  issue_uid             TEXT,                           -- (existing v2)
  issue_number          INTEGER,
  related_issue_id      INTEGER REFERENCES issues(id),
  related_issue_uid     TEXT,                           -- (existing v2)
  type                  TEXT NOT NULL,
  actor                 TEXT NOT NULL,
  payload               TEXT NOT NULL DEFAULT '{}',
  created_at            DATETIME NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  CHECK (length(trim(actor)) > 0),
  CHECK (json_valid(payload)),
  CHECK (length(uid) = 26),
  CHECK (length(origin_instance_uid) = 26)
);
-- Note: events does NOT carry project_uid as a column. The wire envelope's
-- project_uid field is derived at API-handler time via JOIN to projects.uid
-- (current v2 behavior; unchanged in v3). project_identity and project_id
-- already let consumers resolve to the project; denormalizing project_uid
-- onto every event row would be ~26 bytes × every event for a JOIN that
-- runs once per envelope on the API path. If sync ever proves the JOIN
-- expensive, additive cutover later.

-- purge_log: add uid (NOT NULL UNIQUE) and origin_instance_uid (NOT NULL)
CREATE TABLE purge_log (
  id                          INTEGER PRIMARY KEY AUTOINCREMENT,
  uid                         TEXT NOT NULL UNIQUE,     -- NEW (v3)
  origin_instance_uid         TEXT NOT NULL,            -- NEW (v3)
  project_id                  INTEGER NOT NULL,
  project_uid                 TEXT,                     -- (existing v2)
  purged_issue_id             INTEGER NOT NULL,
  issue_uid                   TEXT,                     -- (existing v2)
  project_identity            TEXT NOT NULL,
  -- ... (other columns unchanged)
  CHECK (length(trim(actor)) > 0),
  CHECK (length(uid) = 26),
  CHECK (length(origin_instance_uid) = 26)
);
```

New indexes:

```sql
-- events.uid is already covered by the UNIQUE constraint's auto-index.
CREATE INDEX idx_events_origin_instance ON events(origin_instance_uid);
-- purge_log.uid auto-indexed by UNIQUE.
CREATE INDEX idx_purge_log_origin_instance ON purge_log(origin_instance_uid);
```

The `events.uid` and `purge_log.uid` UNIQUE constraints produce automatic `sqlite_autoindex_*` indexes. The origin indexes support the future sync protocol's "give me events from origin X since cursor Y" query shape.

### 5.2 First-init and cutover behavior for `meta.instance_uid`

**Single ownership rule: `db.Open` is the only writer of `meta.instance_uid` outside an import transaction.** `0001_init.sql` does not seed it. After running migrations, `db.Open` checks whether `meta.instance_uid` exists; if not, it generates one via `uid.New()` and writes it. Because `db.Open` is also the entry point used by tests, by the import target's fresh-DB initialization, and by the cutover's `<path>.import.tmp.db` step, this single rule guarantees every current-schema DB has a valid `meta.instance_uid` immediately after open.

What different paths do with that initial value:

- **Fresh daemon init** (no prior DB): `db.Open` writes a fresh `meta.instance_uid`. The daemon serves with that identity. Subsequent boots: the row already exists; `db.Open` is a no-op for this key.

- **v2→v3 cutover**: The cutover's `db.Open(<path>.import.tmp.db)` writes a fresh `meta.instance_uid` (call it `LOCAL`). The import transaction then uses `LOCAL` as the `origin_instance_uid` for every event and purge_log row backfilled from the v2 source. The atomic swap installs the temp DB at `<path>`; the daemon serves with that identity.

- **`kata import` of a v3 export, default mode**: `db.Open(<target>)` writes a fresh `meta.instance_uid`. The import transaction then **overwrites** that value with the source's `meta.instance_uid` from the export's meta records (`UPDATE meta SET value = ? WHERE key = 'instance_uid'`). End result: imported DB carries the source's identity. This is the backup/restore path — same logical kata installation.

- **`kata import --new-instance`**: `db.Open(<target>)` writes a fresh `meta.instance_uid`. The import transaction **skips** the source's `meta.instance_uid` meta record (treats it as an unknown key for this mode). End result: imported DB has a fresh local identity. Imported `events.origin_instance_uid` and `purge_log.origin_instance_uid` columns are **NOT rewritten** — they preserve the original origins (loop-detection needs this: those events were authored by the cloned-from instance, not by the new clone).

The default vs. `--new-instance` distinction: default preserves identity end-to-end (backup/restore); `--new-instance` preserves the *history's* origins but starts a new local identity (clone to new machine). Both override paths happen inside the import transaction so a failed import rolls back the override too.

This consolidates the previous draft's "daemon does it after `db.Open`" and "`db.Open` does it" into one rule: **`db.Open` always sets it; import options mutate it in-transaction.** Tests that call `db.Open` get a valid `meta.instance_uid` for free.

### 5.3 v2→v3 cutover fill rules

When the importer processes a JSONL stream whose `meta.export_version` is `2`, for each record kind that gained a field in v3 the importer applies the rules below. **What is and isn't deterministic across reruns is intentional and called out per-rule:**

- **`meta.instance_uid`** — generated **freshly with `uid.New()`** by the destination's `db.Open` step (per §5.2) before the import transaction begins. This is **deliberately non-deterministic across reruns**: two clones of the same legacy v2 DB are two different kata installations and must have distinct identities. Call the per-run value `LOCAL`.
- **`events.uid`** — `uid.FromStableSeed([]byte(fmt.Sprintf("event:%d:%d", events.project_id, events.id)), events.created_at)`. **Deterministic per source row across reruns** — the seed and timestamp are stable.
- **`events.origin_instance_uid`** — `LOCAL`. Inherits non-determinism from `meta.instance_uid` by design: a re-cutover from the same v2 source produces a different `LOCAL` and therefore different `origin_instance_uid` on every backfilled event. This is correct: a re-cutover is a new instance.
- **`purge_log.uid`** — `uid.FromStableSeed([]byte(fmt.Sprintf("purge:%d:%d", purge_log.project_id, purge_log.id)), purge_log.purged_at)`. **Deterministic per source row across reruns.**
- **`purge_log.origin_instance_uid`** — `LOCAL`. Same non-determinism as events.

Backfilling `origin_instance_uid` with the destination's `LOCAL` (rather than NULL or some "pre-federation" marker) means every v3 record has a valid origin from day one. There is no "pre-federation" branch in any future sync consumer.

The seed shapes (`event:<project_id>:<id>`, `purge:<project_id>:<id>`) are part of the v2→v3 fill contract: changing them on a re-run would re-issue row UIDs (the deterministic ones above).

**Determinism boundary, summarized:** row UIDs that go through `FromStableSeed` (`projects.uid`, `issues.uid`, `events.uid`, `purge_log.uid`) ARE deterministic across reruns of the same source. `meta.instance_uid` and the `origin_instance_uid` columns derived from it are NOT — they are intentionally fresh on every cutover so two clones of the same legacy DB become two distinct instances.

## 6. JSONL contract changes (`internal/jsonl/`)

All v3-additive. v2 exports (from a v2 binary) are unaffected. v3 exports include the new fields.

### 6.1 New `meta` key

```json
{"kind":"meta","data":{"key":"instance_uid","value":"01J..."}}
```

Emitted by v3 exporters as part of the meta-records prelude (after `export_version`, alongside `schema_version` / `created_by_version`). Consumed by v3 importers per §5.2.

### 6.2 `event` envelope additions

```jsonc
{"kind":"event","data":{
  "id": 81235,
  "uid": "01J...",                       // NEW (v3)
  "origin_instance_uid": "01J...",       // NEW (v3)
  "project_id": 1, "project_identity": "...",
  "issue_id": 110, "issue_number": 2, "issue_uid": "01J...",
  "related_issue_id": null, "related_issue_uid": null,
  "type": "issue.commented", "actor": "wesm",
  "payload": {"comment_id": 12},
  "created_at": "..."
}}
```

(`project_uid` is not in the row, so it's not in the JSONL envelope either. The API's `EventEnvelope` adds it via JOIN at handler time; that's an API-shape decision unaffected by this spec.)

### 6.3 `purge_log` envelope additions

```jsonc
{"kind":"purge_log","data":{
  "id": 1,
  "uid": "01J...",                       // NEW (v3)
  "origin_instance_uid": "01J...",       // NEW (v3)
  "project_id": 1, "project_uid": "01J...", "purged_issue_id": 42,
  "issue_uid": "01J...", "project_identity": "...",
  // ... (all existing v2 fields)
}}
```

### 6.4 Importer behavior

- v3 source → v3 binary: straight import. `meta.instance_uid` preserved; events/purge_log carry their UIDs and origins verbatim.
- v3 source → v3 binary with `--new-instance`: target already has a `meta.instance_uid` from `db.Open`; importer skips the source's record so the target keeps its own identity. Events/purge_log keep their UIDs and original `origin_instance_uid` (per §5.2).
- v2 source → v3 binary: upgrade import (cutover path). Target already has a `meta.instance_uid` from `db.Open` of the temp DB; importer applies §5.3 fill rules using that value as `LOCAL`.
- v1 source → v3 binary: upgrade import that combines v1→v2 (per issue-identity spec §5.2) and v2→v3 (this spec §5.3) fill rules in one pass. Test coverage in §8.5.

## 7. Wire envelopes (`internal/api/types.go` + Plan 4 SSE)

`EventEnvelope` (Plan 4 §4.1) gains:

```go
type EventEnvelope struct {
    EventID            int64           `json:"event_id"`
    EventUID           string          `json:"event_uid"`              // NEW (v3)
    OriginInstanceUID  string          `json:"origin_instance_uid"`    // NEW (v3)
    Type               string          `json:"type"`
    // ... (existing v2 fields stay)
}
```

`PurgeLogEntry` gains `uid` and `origin_instance_uid`. Existing fields stay.

A new endpoint surfaces the local instance identity:

```
GET /api/v1/instance
→ 200 {"instance_uid": "01J..."}
```

This is the smallest possible surface that lets a future spoke discover its hub's identity (or any kata daemon discover any other's). The sync protocol design will compose richer endpoints on top — but that's the only API surface this spec adds.

## 8. Test plan

### 8.1 Unit — `internal/uid` (no new tests needed)

The library is unchanged. v3 uses the same `uid.New()` and `uid.FromStableSeed()` helpers as v2.

### 8.2 Unit — schema invariants (`internal/db/schema_completeness_test.go`)

- After a fresh init: `meta.instance_uid` exists, value is a valid 26-char ULID.
- After a fresh init: `currentSchemaVersion` is `3`; `meta.schema_version` is `'3'`.
- A second daemon boot does not regenerate `meta.instance_uid` — the value is stable across restarts.
- `events.uid` and `events.origin_instance_uid` are both NOT NULL on any inserted row (constraint test: insert with NULL fails).
- Same for `purge_log.uid` and `purge_log.origin_instance_uid`.

### 8.3 Unit — daemon insert paths

- A new event written via the daemon's mutation handlers carries a fresh `uid` (passes `uid.Valid`) and an `origin_instance_uid` matching `meta.instance_uid`.
- Same for a `purge_log` row produced by `kata purge`.
- The daemon reads `meta.instance_uid` once at startup; mutating the row mid-run (via direct SQL) does not change subsequent insert behavior in the same daemon process. (Documents the "set-once" contract.)

### 8.4 v2→v3 cutover (`internal/jsonl/cutover_test.go` extension)

Build a v2 fixture DB with curated rows covering events (varied types, idempotency-key payloads) and a `purge_log` row with the synthetic `purge_reset_after_event_id`.

- After cutover: `meta.instance_uid` is set, valid ULID.
- Every event has `uid` (valid ULID, unique within the table) and `origin_instance_uid` matching `meta.instance_uid`.
- Every `purge_log` row has `uid` and `origin_instance_uid` populated, both matching the new instance identity.
- The `purge_reset_after_event_id` reservation is preserved (per JSONL spec §5.2 — same property as v2 cutover).
- Re-running the cutover on the same v2 source produces identical `events.uid` and `purge_log.uid` values across runs (deterministic via `FromStableSeed`).

### 8.5 v1 → v3 cutover (combined)

A v1 source fixture (no UIDs at all) cutting over directly to v3.

- Target's `meta.instance_uid` is already set by `db.Open` of the temp DB (per §5.2); importer applies v1→v2 fill rules (issue-identity spec §5.2) and v2→v3 fill rules (this spec §5.3) in one pass.
- After cutover: every project has `uid`; every issue has `uid`; every link has matching `from_issue_uid` / `to_issue_uid`; every event has `uid` + `origin_instance_uid`; every purge_log row has `uid` + `origin_instance_uid`.
- Determinism: re-running on the same v1 source produces identical row UIDs (`projects.uid`, `issues.uid`, `events.uid`, `purge_log.uid`) — these all flow through `FromStableSeed`. `meta.instance_uid` and the `origin_instance_uid` columns derived from it are **intentionally fresh on every cutover** (per §5.3), so the re-run assertion explicitly excludes those fields.

### 8.6 Roundtrip — v3 export → v3 import preserves identity

`TestRoundtrip_V3_PreservesInstanceUID`:

- Fresh init at v3 → 4 projects, 10 issues, sync state.
- `kata export` to JSONL.
- `kata import` (default mode) into a fresh tmpdir.
- Assert `meta.instance_uid` matches between source and destination.
- Assert every event's `origin_instance_uid` matches the source's `meta.instance_uid`.

### 8.7 `kata import --new-instance`

`TestImport_NewInstance_RegeneratesIdentity`:

- v3 export from source A (with `meta.instance_uid = INST_A`).
- Import with `--new-instance` into a fresh tmpdir.
- Assert destination's `meta.instance_uid != INST_A` and is a valid ULID.
- Assert imported events' `origin_instance_uid == INST_A` (preserved, not rewritten).

### 8.8 API endpoint

- `GET /api/v1/instance` returns `{"instance_uid": "<value>"}` matching `meta.instance_uid`.
- Calling before the meta row is set (only possible during init bootstrap, theoretical in production) returns 503 `instance_uid_unset`.

### 8.9 e2e

`TestSmoke_FederationFoundationV3`:

1. Start daemon at v3 against an empty DB.
2. Hit `GET /api/v1/instance`; capture the instance UID.
3. Create a project, an issue, a comment.
4. SSE-tail the events endpoint; capture the event envelopes.
5. Assert each event's `origin_instance_uid` equals the captured instance UID.
6. Purge the issue; assert the resulting `purge_log` row has `uid` and `origin_instance_uid` populated.
7. `kata export`; assert the JSONL contains the `meta.instance_uid` record and the events have `uid`/`origin_instance_uid`.

## 9. Open questions / tunables

- **`projects.created_by_instance_uid` / `issues.created_by_instance_uid`?** Not added in v3. The sync protocol may want this for "who created this issue" attribution; if so, future cutover. Currently kata has no notion of authorship beyond the `actor` (a free-form string), and projects/issues authorship is implied by their first-event's `origin_instance_uid` once events have one. Keeping the schema minimal for v3.
- **`comments.uid` / `issue_labels.uid`?** Per §1.5, deferred. Comments and labels are addressed via parent-issue + event-uid. If sync design needs row-level comment UIDs, additive cutover.
- **`instances` table?** Not added. Instance identity is a property carried in records, not a referential entity. If sync needs a "known peers" registry, that's a sync-protocol concern, not an identity-layer one.
- **Mutability of `meta.instance_uid`.** Locked-once-written by convention, not by SQL constraint. A SQL trigger could enforce immutability but it would prevent the `--new-instance` import path that legitimately overwrites the row. The CLI/daemon contract is "never mutate this value during a daemon's lifetime"; cutover and clone-import are the only writers.
- **Pre-existing `meta` rows imported.** A v2 export contains `meta` records like `created_by_version`. The v3 importer preserves them. If a v3 export contains an `instance_uid` meta record, default mode overwrites the target's value (written by `db.Open`) with the source's; `--new-instance` skips the source's record so the target keeps the identity `db.Open` wrote. Other meta keys (anything not in `{schema_version, instance_uid, created_by_version, exported_at}`) are preserved verbatim — forward-compat for any future meta-table additions.
- **Should we retire the multi-file migration runner?** With cutover-via-JSONL as the schema-evolution path, the existing `internal/db/migrations/*.sql` runner serves only two functions: (a) apply `0001_init.sql` to a fresh DB; (b) record `meta.schema_version = currentSchemaVersion` after init. Numbered migrations beyond `0001_init.sql` aren't part of the new model — additive, harmless changes (a new index, a CHECK relaxation, a new column with a SQL-expressible default) might still be worth shipping in-place rather than going through a full export/import cycle, but everything that needs backfilling is now a JSONL cutover. Open question: should we (1) leave the runner as-is and treat new files as escape-hatches for harmless edits, (2) collapse the runner to "apply init script, stamp version" and force every change through cutover, or (3) keep the runner but delete the `[]migration` slice plumbing now that no `00NN_*.sql` files exist beyond `0001_init.sql`? Recommend **(3)** — preserve the loader but simplify the data structure to "one canonical init script + version stamp," then revisit (1) vs (2) the first time someone proposes a harmless additive change. Tracked separately from this spec.

## 10. Sequencing

This spec lands **after** the JSONL roundtrip + issue identity work currently committed to `main` is pushed and accepted. Order:

1. Push the existing JSONL + issue-identity commits to remote (they've been validated against the live DB; nothing blocks).
2. Land the federation-foundation work as one commit batch:
